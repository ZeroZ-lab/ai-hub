# Module 4.3 - Skills 最佳实践

> **学习目标**：学会编写可复用、可维护、稳定输出的 Skills。

## 1. 技能范围要窄

✅ 好的 Skill：
- 只做“代码审查”或“需求拆分”
- 输出格式稳定

❌ 不好的 Skill：
- “万能助手”式描述，触发过于宽泛

**建议**：每个 Skill 聚焦一个明确任务。

---

## 2. 触发描述要具体

`description` 决定触发匹配度：

- ✅ "Review code changes, highlight bugs and missing tests"
- ❌ "Help with programming"

让模型更准确识别何时调用。

---

## 3. 输出格式要可执行

输出建议包含：
- 严重级别（Critical/High/Medium/Low）
- 文件/行号
- 可操作的修复建议

**示例**：
```
Findings:
- [High] src/payment.py:42 - Potential double charge due to retry loop.
```

---

## 4. 让 Skill 明确工具使用

在 Skill 中说明“需要读取哪些文件”。
例如：
```
If code scope is unclear, ask for file paths.
Use Read to load files before reviewing.
```

这样能避免“凭空猜测”。

---

## 5. 给出测试建议

审查不仅指出问题，还应提示测试方案：
- 单元测试覆盖边界条件
- 回归测试覆盖行为变化

---

## 6. 常见错误

- 描述过泛导致误触发
- 输出冗长、无法用于 PR 评论
- 没有明确的严重程度和定位

---

## 7. 维护策略

- 将 Skill 版本号写在头部
- 在每次项目规则变更时更新 Skill
- 配合团队 Code Review 规范

---

## 总结

✅ **高质量 Skill 特征**：
1. 范围清晰
2. 触发精准
3. 输出可执行
4. 工具使用明确
5. 与团队规范一致
