# Module 5.1 - MCP 协议概念

> **学习目标**：理解 Model Context Protocol (MCP) 的核心概念、组成与适用场景。

## 为什么需要 MCP？

在前面的模块中，我们通过 SDK 直接注册工具。但当工具越来越多、需要跨项目复用时，会遇到：

- **复用成本高**：每个项目都要重复定义工具
- **运维不集中**：工具更新需要修改多个代码仓库
- **权限难管理**：访问数据源的凭证散落在各处

**MCP** 提供统一的协议层，把工具、资源、提示词封装成服务端能力，供客户端按需调用。

---

## MCP 是什么？

**Model Context Protocol (MCP)** 是一个“模型与外部能力之间的协议”，它定义了：

- 客户端如何发现可用能力
- 服务端如何暴露工具和资源
- 双方如何进行标准化调用与响应

**核心角色**：
- **Client**：模型侧，负责调用 MCP 服务
- **Server**：能力提供方，暴露 tools/resources/prompts

---

## MCP 的核心能力

MCP 主要提供三类能力：

1. **Tools**：可执行操作（调用 API、写文件、计算等）
2. **Resources**：可读取的资源（文件、数据库记录、配置等）
3. **Prompts**：可复用的提示模板

---

## 典型流程

```
用户请求
  ↓
Client 选择 MCP Server
  ↓
Client 调用 tool/resource
  ↓
Server 返回结果
  ↓
Client 生成响应
```

---

## MCP 适合的场景

✅ **适合使用 MCP 的场景**
- 多项目共用的一组工具
- 公司内部统一的数据访问服务
- 与外部服务集成（DB、BI、支付系统）

❌ **不一定需要 MCP 的场景**
- 单一项目的小工具
- 一次性的快速脚本

---

## MCP vs SDK 内置工具

| 对比维度 | SDK 内置工具 | MCP Server |
|----------|--------------|------------|
| 部署 | 与客户端同进程 | 独立服务 |
| 复用 | 项目内 | 跨项目 |
| 管理 | 分散 | 集中 |
| 权限 | 本地管理 | 服务端统一控制 |
